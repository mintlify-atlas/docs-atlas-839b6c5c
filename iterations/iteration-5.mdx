## Critical: Conditional Rendering Pattern

<Note>
**IMPORTANT:** The modal should only render when a product is selected. Never use an `isOpen` prop pattern.

**Correct:**
```typescript
{selectedProduct && (
  <ProductDetail product={selectedProduct} onClose={handleCloseModal} />
)}
```

**Incorrect:**
```typescript
{/* ❌ DON'T DO THIS */}
<ProductDetail 
  product={selectedProduct} 
  isOpen={!!selectedProduct}
  onClose={handleCloseModal}
/>
```

Why? Conditional rendering is simpler, more React-like, and avoids rendering unnecessary DOM nodes.
</Note>

## The Native Dialog Element

<Tip>
The `<dialog>` element is a native HTML element with built-in modal functionality:

```html
<dialog open>
  <!-- Modal content -->
</dialog>
```

**Benefits:**
- Semantic HTML
- Automatic focus management
- Backdrop styling with `::backdrop` pseudo-element
- Accessibility features built-in

**The `open` attribute:** Controls whether the dialog is visible.
</Tip>

## Implementation Hints

<Accordion title="How do I make a component clickable?">

Add an `onClick` handler to the root element:

```typescript
interface ProductCardProps {
  product: Product
  onClick?: () => void
}

<article onClick={onClick}>
  {/* ... */}
</article>
```

Make it clear to users by adding a cursor style:
```typescript
style={{ cursor: onClick ? 'pointer' : 'default' }}
```

</Accordion>

<Accordion title="How do I manage modal state?">

Store the selected product in state:

```typescript
const [selectedProduct, setSelectedProduct] = useState<Product | null>(null)

// Open modal
const handleProductClick = (product: Product) => {
  setSelectedProduct(product)
}

// Close modal
const handleCloseModal = () => {
  setSelectedProduct(null)
}
```

When `selectedProduct` is `null`, the modal isn't rendered.

</Accordion>

<Accordion title="How do I add keyboard event listeners?">

Use `useEffect` to add and clean up event listeners:

```typescript
useEffect(() => {
  const handleKeyDown = (event: KeyboardEvent) => {
    if (event.key === 'Escape') {
      onClose()
    }
  }
  
  window.addEventListener('keydown', handleKeyDown)
  
  // Cleanup function runs when component unmounts
  return () => window.removeEventListener('keydown', handleKeyDown)
}, [onClose])
```

Always include `onClose` in the dependency array.

</Accordion>

<Accordion title="Why include onClose in the dependency array?">

ESlint's `exhaustive-deps` rule requires all values used inside `useEffect` to be listed in the dependency array. This ensures:

1. The effect re-runs if `onClose` changes
2. The cleanup function removes the correct listener
3. No stale closures

```typescript
useEffect(() => {
  const handleKeyDown = (event: KeyboardEvent) => {
    if (event.key === 'Escape') {
      onClose() // Uses onClose
    }
  }
  window.addEventListener('keydown', handleKeyDown)
  return () => window.removeEventListener('keydown', handleKeyDown)
}, [onClose]) // Must include onClose
```

</Accordion>

## Common Issues

<Tip>
**Issue:** "Cannot read property 'displayName' of null"

**Solution:** Use conditional rendering:

```typescript
{selectedProduct && (
  <ProductDetail product={selectedProduct} onClose={handleCloseModal} />
)}
```

The modal component only renders when `selectedProduct` is not null.
</Tip>

<Tip>
**Issue:** ESC key closes modal but warning appears

**Solution:** Make sure to include `onClose` in the `useEffect` dependency array:

```typescript
useEffect(() => {
  // ...
}, [onClose]) // Don't forget!
```
</Tip>

<Tip>
**Issue:** Clicking product card scrolls instead of opening modal

**Solution:** Make sure you're passing the `onClick` prop to the `<article>` element, not to a child element.
</Tip>

<Tip>
**Issue:** Modal appears but overlay doesn't cover whole screen

**Solution:** The overlay needs to be positioned fixed with full viewport dimensions. Check the CSS:

```css
.product-detail__overlay {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  bottom: 0;
  background: rgba(0, 0, 0, 0.5);
  z-index: 999;
}
```
</Tip>
